2026-02-28 — Task 1.1.4: Create nullable type utilities (complete)

Created nullable type utilities for C# code generation in src/utils/nullable.ts.

- **Key deliverables:**
  - `isPropertyNullable(property)` — determines if a property should render as T? in C#
  - `unwrapNullableType(type)` — strips SdkNullableType wrapper for TypeExpression
  - `isCollectionType(type)` — checks array/dict (collections are never nullable)
  - 26 unit tests in test/nullable.test.ts covering all type categories

- **Design decision:** Three separate functions (not a single combined resolver) for composability. The property nullable check accepts any `{ type: SdkType; optional: boolean }` shape, not just SdkModelPropertyType, for flexibility across property-like contexts.

- **Rules implemented (matching legacy PropertyProvider.cs lines 86-88):**
  - Optional non-collection types → nullable (both value and reference types, matching legacy)
  - Required types → not nullable
  - Collections (array/dict) → never nullable, even when wrapped in SdkNullableType
  - Explicitly nullable types (SdkNullableType wrapper) → nullable (unless collection)

- **TCGC insight:** SdkNullableType is TCGC's representation of `T | null`. It wraps the inner type. The `optional` flag on properties is orthogonal — a property can be optional without SdkNullableType, or have SdkNullableType without being optional.

- **Usage pattern for downstream tasks (1.2.2 model properties):**
  ```tsx
  const innerType = unwrapNullableType(prop.type);
  const nullable = isPropertyNullable(prop);
  <Property type={<TypeExpression type={innerType} />} nullable={nullable} />
  ```

2026-03-01 — Task 11.1.5: Handle inline string literal union types in TypeExpression (complete)

Problem: The emitter crashed with "Unsupported type for TypeExpression: Union ("red" | "blue")" when a model property used an inline string literal union type. The upstream TypeExpression only handles named unions and nullable unions, but inline literal unions are unnamed.

Root cause: TCGC converts inline string literal unions to SdkEnumType, and the emitter generates enum files for them. But when ModelProperty passes type.__raw! to TypeExpression, the raw type is an unnamed TypeSpec Union, which TypeExpression doesn't know how to handle.

Fix: Added a `forTypeKind("Union", ...)` override in CSharpTypeExpression.tsx using the existing Experimental_ComponentOverridesConfig pattern. The override:
- Delegates named unions and nullable unions to the default TypeExpression
- For unnamed non-nullable unions (inline literal unions), emits a <Reference refkey={efCsharpRefkey(union)} /> that resolves to the generated enum declaration

Key design decision: Used the override mechanism rather than fixing individual callsites (ModelProperty, serialization, etc.) because:
1. Centralizes the fix for ALL TypeExpression usages
2. Follows the existing pattern (scalar and intrinsic overrides are already there)
3. The enum file declarations already register efCsharpRefkey(rawType) as their refkey

Files changed:
- src/components/CSharpTypeExpression.tsx — Added Union override and hasNullVariant helper
- test/model.test.ts — Added test for inline string literal union type resolution

Pattern: The override mechanism in CSharpTypeExpression.tsx now handles three type kinds: Union, Scalar, and Intrinsic. Future type expression overrides should follow the same pattern.

## 2026-03-02 — Task 3.3.1: Create RestClientFile component (complete)

### What was done
Created `src/components/clients/RestClientFile.tsx` — generates `{ClassName}.RestClient.cs` partial class files with:
- Static `PipelineMessageClassifier` fields with lazy `??=` initialization per unique set of success status codes
- `Create{Op}Request` methods for each HTTP operation on the client

### Method body pattern
Each CreateRequest method follows the System.ClientModel pattern:
1. `ClientUriBuilder uri = new ClientUriBuilder();` + `uri.Reset(_endpoint);`
2. Path segments via `uri.AppendPath(literal, false)` and `uri.AppendPath(param, true)`
3. Query params via `uri.AppendQuery(serializedName, value, true)` with null checks for optional
4. `PipelineMessage message = Pipeline.CreateMessage(uri.ToUri(), "VERB", classifier);`
5. `PipelineRequest request = message.Request;`
6. Custom headers → Content-Type (from body) → Accept (from responses)
7. `request.Content = content;` (if body)
8. `message.Apply(options);` + `return message;`

### Key patterns and decisions
- **Partial class**: Uses `namekey(className, { ignoreNameConflict: true })` — same pattern as ModelSerializationFile — to avoid Alloy's symbol deduplication renaming
- **Type mapping**: Direct switch-based mapping from SdkType.kind to C# types (not TypeExpression, which expects TypeSpec `Type` not SdkType). Returns strings for keywords (string, int), refkeys for types needing `using` (DateTimeOffset, Uri, BinaryData)
- **Parameter ordering**: path (0) → required header/query (100) → body (200/300) → optional header/query (400) → RequestOptions (always last)
- **ClientUriBuilder/TypeFormatters**: Raw strings, not builtins — they're internal types in the same namespace
- **Constants**: Params with `type.kind === "constant"` are hardcoded in method body, not method params
- **Client fields**: Params with `onClient: true` (apiVersion, etc.) use class field references like `_apiVersion`
- **Content-Type**: Derived from `bodyParam.defaultContentType`, not a method parameter
- **Accept**: Derived from response `defaultContentType`/`contentTypes`

4. **Reordered override logic** — nullable unions checked first (delegated to default), then multi-type (→ BinaryData), then named extensible enums (→ default), then unnamed extensible enums (→ efCsharpRefkey).

## 2026-03-04: Task 12.2.10 — Fix union float/int literal property types in optionality spec

**Problem**: TCGC preserves numeric literal union values verbatim as enum member names (e.g., "1.25", "2.375"). The Alloy `pascalCase` transform from `change-case` converts "1.25" → "1_25" (word at index>0 starting with digit gets `_` prefix, but at the wrong position), and "1" → "1" (unchanged). Both are invalid C# identifiers.

**Root cause**: `change-case` v5's `pascalCaseTransformFactory` adds `_` before digits only at word index > 0: `"1_25"` comes from words `["1", "25"]` where the second word gets `_` prefix. The legacy emitter's `ToIdentifierName()` instead prefixes the WHOLE result with `_` and removes non-alphanumeric chars.

**Solution**: Added `fixedEnumMemberName()` utility function exported from `FixedEnumFile.tsx`:
- If `rawName` starts with a digit: `_${rawName.replace(/[^a-zA-Z0-9]/g, "")}`
  - "1.25" → "_125", "2.375" → "_2375", "1" → "_1"
- Otherwise: delegates to `namePolicy.getName(rawName, "enum-member")`

Also removed `<EnumMember>` component and replaced with raw text output, since:
- `EnumMember` internally applies `pascalCase` which can't produce `_125`
- No refkeys to fixed enum members exist anywhere else in the codebase

**Files changed**: `src/components/enums/FixedEnumFile.tsx`, `src/components/enums/FixedEnumSerializationFile.tsx`, `test/fixed-enum.test.ts`, `test/fixed-enum-serialization.test.ts`

**Tests added**: 2 new tests for numeric-starting enum member names in declaration and serialization files.
