2026-02-28 — Task 1.1.4: Create nullable type utilities (complete)

Created nullable type utilities for C# code generation in src/utils/nullable.ts.

- **Key deliverables:**
  - `isPropertyNullable(property)` — determines if a property should render as T? in C#
  - `unwrapNullableType(type)` — strips SdkNullableType wrapper for TypeExpression
  - `isCollectionType(type)` — checks array/dict (collections are never nullable)
  - 26 unit tests in test/nullable.test.ts covering all type categories

- **Design decision:** Three separate functions (not a single combined resolver) for composability. The property nullable check accepts any `{ type: SdkType; optional: boolean }` shape, not just SdkModelPropertyType, for flexibility across property-like contexts.

- **Rules implemented (matching legacy PropertyProvider.cs lines 86-88):**
  - Optional non-collection types → nullable (both value and reference types, matching legacy)
  - Required types → not nullable
  - Collections (array/dict) → never nullable, even when wrapped in SdkNullableType
  - Explicitly nullable types (SdkNullableType wrapper) → nullable (unless collection)

- **TCGC insight:** SdkNullableType is TCGC's representation of `T | null`. It wraps the inner type. The `optional` flag on properties is orthogonal — a property can be optional without SdkNullableType, or have SdkNullableType without being optional.

- **Usage pattern for downstream tasks (1.2.2 model properties):**
  ```tsx
  const innerType = unwrapNullableType(prop.type);
  const nullable = isPropertyNullable(prop);
  <Property type={<TypeExpression type={innerType} />} nullable={nullable} />
  ```

2026-03-01 — Task 11.1.5: Handle inline string literal union types in TypeExpression (complete)

Problem: The emitter crashed with "Unsupported type for TypeExpression: Union ("red" | "blue")" when a model property used an inline string literal union type. The upstream TypeExpression only handles named unions and nullable unions, but inline literal unions are unnamed.

Root cause: TCGC converts inline string literal unions to SdkEnumType, and the emitter generates enum files for them. But when ModelProperty passes type.__raw! to TypeExpression, the raw type is an unnamed TypeSpec Union, which TypeExpression doesn't know how to handle.

Fix: Added a `forTypeKind("Union", ...)` override in CSharpTypeExpression.tsx using the existing Experimental_ComponentOverridesConfig pattern. The override:
- Delegates named unions and nullable unions to the default TypeExpression
- For unnamed non-nullable unions (inline literal unions), emits a <Reference refkey={efCsharpRefkey(union)} /> that resolves to the generated enum declaration

Key design decision: Used the override mechanism rather than fixing individual callsites (ModelProperty, serialization, etc.) because:
1. Centralizes the fix for ALL TypeExpression usages
2. Follows the existing pattern (scalar and intrinsic overrides are already there)
3. The enum file declarations already register efCsharpRefkey(rawType) as their refkey

Files changed:
- src/components/CSharpTypeExpression.tsx — Added Union override and hasNullVariant helper
- test/model.test.ts — Added test for inline string literal union type resolution

Pattern: The override mechanism in CSharpTypeExpression.tsx now handles three type kinds: Union, Scalar, and Intrinsic. Future type expression overrides should follow the same pattern.
