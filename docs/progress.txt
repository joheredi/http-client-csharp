## 2026-02-27 — Task 0.3.4: Implement license header rendering
**What was done:**
- Created `src/utils/header.ts` with `getLicenseHeader(options)` function
- The function generates the standard C# file header matching the legacy emitter's golden file format
- Supports three modes: (1) raw `header` string with `// ` prefix per line, (2) constructed from `company` + `name` fields, (3) minimal header with no license info
- Always includes `// <auto-generated/>` marker and `#nullable disable` directive
- Created `test/header.test.ts` with 9 tests covering all configurations and edge cases

**Design decisions:**
- Utility function approach (not a JSX component) since header is static text that works directly as SourceFile's `header` prop Children
- Raw `header` field lines get `// ` prefix (matches legacy CodeWriter.ToString behavior)
- When no raw header is provided, constructs from `company` and `name` fields
- When no license config at all, emits only auto-generated marker and nullable disable

**Patterns used:**
- Returns a `\n`-joined string compatible with Alloy SourceFile `header` prop
- Downstream usage: `<SourceFile header={getLicenseHeader(options)}>...</SourceFile>`

**For next iterations:**
- This unblocks 1.4.1 (FixedEnumFile), 1.6.1 (ExtensibleEnumFile), and 1.2.1 (ModelFile, which also needs 1.1.2)
- The `header` prop on Alloy SourceFile renders content before `using` directives with an automatic blank line separator
- Do NOT use `<Prose>` for the header — it collapses whitespace and word-wraps, destroying the exact format

- Component uses @alloy-js/csharp EnumDeclaration + EnumMember directly (not emitter-framework wrapper)
- Doc comments use TCGC `summary` → `doc` → `{name}.` fallback chain matching legacy DocHelpers

- Wired into emitter.tsx: filters `sdkContext.sdkPackage.enums` by `isFixed` and renders FixedEnumFile for each
- Created `src/components/enums/index.ts` barrel export
- Updated `src/components/index.ts` to export FixedEnumFile
- Added `HttpTester` to `test/test-host.ts` with `.importLibraries()` for HTTP decorator support
- Created `test/fixed-enum.test.ts` with 7 tests covering: enum generation, header, doc comments, namespace, no values for string-backed, no output when empty, @doc decorator

**Design decisions:**
- Used @alloy-js/csharp EnumDeclaration directly instead of emitter-framework wrapper because:
  1. Need control over doc comment generation (TCGC fallback to member name)
  2. Need future extensibility for int-backed enum values (EnumMember lacks value prop)
  3. TCGC types are our primary data source, not TypeSpec types
- Single-line doc comment format `/// <summary> ... </summary>` via raw strings (matching golden file)
- Header as children (not prop) because @alloy-js/csharp SourceFile doesn't expose a `header` prop

**Patterns used:**
- `useCSharpNamePolicy()` for enum and member naming
- `refkey(sdkType)` with TCGC objects as seeds for stable cross-file references
- `<For each={...} joiner={",\n"}>` for comma-separated enum members
- `HttpTester = Tester.importLibraries()` to enable `using TypeSpec.Http;` in tests

**Known issues:**
- Namespace brace style is K&R (`namespace N {`) instead of Allman (`namespace N\n{`). This is an Alloy Block formatting default — not controllable at component level.
- @alloy-js/csharp SourceFile doesn't have a `header` prop. Header is injected as raw text children before Namespace.
